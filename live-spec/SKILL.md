# live-spec Skill

## 核心理念

**界面即文档，文档即界面——两者是同一事物的两面，永远同步。**

每次生成或修改页面，必须同步输出/更新对应的 `live-spec.ts`，
并确保页面中已挂载 `<LiveSpec>` 悬浮球组件。

---

## 触发时机

以下任意情况均触发本 Skill 的完整工作流：

- 用户要求新建一个页面或模块
- 用户要求修改/完善已有页面的界面结构
- 用户明确说"生成需求文档"、"更新需求说明"、"同步需求"、"加悬浮球"
- 用户使用 `/live-spec <路径>` 处理指定页面，例如 `/live-spec app/dashboard`
- 用户使用 `/live-spec all` 批量处理项目所有页面
- 用户使用 `/live-spec init` 仅初始化悬浮球组件，不生成需求文档
- 用户使用 `/live-spec tag` 或 `/live-spec tag "备注"` 打版本快照
- 用户使用 `/live-spec help` 查看所有可用命令

---

## 工作流

### Step 1：确保 LiveSpec 组件存在

检查项目中是否存在 `components/LiveSpec.tsx`（或 `src/components/LiveSpec.tsx`）。

**若不存在**，创建该文件，内容完整照搬 `references/LiveSpec.tsx.template`，不做任何修改直接写入。

**若已存在**，检查文件末尾是否包含注释 `// live-spec skill`——
- 包含：说明是新版，跳过
- 不包含：说明是旧版，**覆盖写入新版**，并告知用户已自动升级

`/live-spec init`：仅执行此步，安装 LiveSpec 组件但不生成任何需求文档，适合在项目初期提前部署悬浮球。

---

### Step 2：收集并融合三路信息源

需求文档的内容来自以下三个来源，按优先级从高到低排列：

#### 来源 A：对话上下文中的明确需求（优先级最高）

回溯当前对话的完整历史，提取所有与该页面/功能相关的信息：
- 用户描述的功能意图、业务逻辑、使用场景
- 用户提出的字段要求、交互规则、特殊约束
- 用户对某个功能的补充说明或修改指令
- 之前轮次中双方确认过的方案细节

**这些信息即使没有体现在源码里，也必须写入需求文档。**
源码是实现，对话是意图——意图比实现更权威。

#### 来源 B：源码中可推断的结构（优先级次之）

读取目标页面目录下的相关文件：
- `page.tsx` / `page.jsx`
- 该页面引用的子组件（同目录或 `components/` 下）
- 相关的 types、constants、api 文件（如有）

从源码推断，**必须穷举当前页面内所有可见或可触达的业务内容**，包括：
- 页面包含哪些主要区域（对应一级编号）
- 每个区域有哪些字段、按钮、交互行为（对应二级/三级编号）
- 表格列的名称、操作行为
- 数据流向（从哪里来、交互后做什么）

**弹窗/抽屉/面板必须深入描述内部内容，不得只写触发入口：**

凡是在当前页面内触发的任何覆盖层（Modal、Drawer、Popover、Dialog、Sheet、
下拉面板、右键菜单等），无论由按钮、表格行操作、图标还是其他元素触发，
都属于当前页面的一部分，必须在触发入口的下级编号中完整描述其内部结构：
- 弹窗/抽屉的标题和用途
- 内部所有表单字段（字段名、类型、校验规则、默认值）
- 内部所有操作按钮及其行为
- 提交/确认后的数据流向和页面反应
- 多步骤流程需按步骤逐一描述

**示例：不能只写**
```
1.1 【新增】
    点击打开【新增弹窗】
```
**必须写成**
```
1.1 【新增】
    点击打开【新增弹窗】
  1.1.1 【新增弹窗】
      用于新增一条记录
      1.1.1.1 【名称】
          文本输入框，必填，最多 64 字符
      1.1.1.2 【类型】
          下拉单选，选项来自枚举值，必填
      1.1.1.3 【描述】
          多行文本，选填
      1.1.1.4 【取消】
          关闭弹窗，不保存
      1.1.1.5 【确认】
          提交表单，校验通过后保存并刷新列表
```

**排除规则（以下内容不得写入需求文档）：**
- `LiveSpec` 组件本身及其悬浮球、抽屉、复制按钮等 live-spec 自身 UI
- 任何 `NODE_ENV === 'development'` 条件下才渲染的调试组件
- 跳转到其他页面的链接（目标页面在对应页面的文档中描述）
- import 语句、类型定义、工具函数等非业务代码

#### 来源 C：合理的功能推断（优先级最低）

当某个交互细节在对话和源码中均未明确，但根据常识或同类功能可以合理推断时，
补充该细节并在描述中用「默认」「通常」等词标注，表示这是推断而非明确需求。

#### 融合规则

| 情况 | 处理方式 |
|------|---------|
| 对话有、源码也有，内容一致 | 正常写入 |
| 对话有、源码没有 | 以对话为准写入，这是意图未落地的部分 |
| 源码有、对话没有 | 从源码提取写入 |
| 对话与源码有冲突 | 以对话为准，源码可能是未完成的实现 |
| 两者都没有但可合理推断 | 写入并标注「默认」 |

### Step 3：生成需求文档内容

严格遵循以下规范生成需求文档字符串，每一条都是硬性约束，不得违反。

#### 结构规则
- 按人看界面的视觉顺序（从上到下、从左到右）组织层级
- 一级编号（1、2、3）对应页面主元素（标题、工具栏、表格…）
- 二级编号（1.1、1.2）对应元素内的子功能或字段
- 三级编号（1.1.1）用于需要进一步拆分的交互细节
- 不描述样式（颜色、字号、间距、宽度）
- 不描述布局位置（由设计规范统一管控）
- 只描述：内容、数据、交互逻辑
- 功能名、按钮名、页面名一律用【】标注

#### 空行规则
- 一级编号条目之间空一行
- 一级编号内部（二级、三级条目）不空行

#### 缩进规则（精确空格数，不得偏差）

| 层级 | 编号行缩进 | 描述行缩进 |
|------|-----------|-----------|
| 1 级 | 0 个空格  | 2 个空格  |
| 2 级 | 2 个空格  | 4 个空格  |
| 3 级 | 4 个空格  | 4 个空格（与编号同级） |

#### 标准输出示例（严格对照此格式）

```
1 【页面标题区】
  显示当前页面名称，右侧展示操作入口

1.1 【新增】
    点击打开【新增弹窗】
1.2 【批量删除】
    勾选记录后激活，点击弹出二次确认

2 【筛选工具栏】
  提供多维度筛选条件，变更后自动刷新列表

2.1 【关键词搜索】
    文本输入框，对名称字段进行模糊匹配
2.2 【状态筛选】
    下拉单选，选项来自接口枚举值，默认「全部」

3 【数据列表】
  展示记录列表，支持分页、排序、行内操作

3.1 【名称列】
    显示记录名称，固定左侧，表头支持文本搜索
    点击名称跳转至【详情页】
3.2 【操作列】
    固定右侧，包含：【编辑】【删除】
    3.2.1 【编辑】
        点击打开【编辑弹窗】，回填当前行数据
    3.2.2 【删除】
        点击弹出二次确认，确认后删除并刷新列表

4 【分页器】
  包含条数统计、每页条数切换（10/20/50）、页码导航、跳转输入框
```

**自检清单**（生成后逐条核对，不通过则重新生成）：
- [ ] 一级条目之间有且只有一个空行
- [ ] 二级、三级条目之间没有空行
- [ ] 一级描述行是 2 个空格缩进
- [ ] 二级编号是 2 个空格缩进，描述行是 4 个空格缩进
- [ ] 三级编号是 4 个空格缩进，描述行是 4 个空格缩进
- [ ] 所有功能名/按钮名/页面名都有【】标注
- [ ] 没有出现任何样式描述（颜色、字号、间距、宽度）
- [ ] 没有出现任何布局位置描述（左侧、右侧、顶部等绝对位置词）
- [ ] 页面内每一个弹窗/抽屉/面板都有对应的内部结构描述，没有只写"点击打开XX弹窗"而无下级内容的条目
- [ ] LiveSpec 组件及其 UI 没有出现在需求文档中

### Step 4：写入页面 live-spec.ts 并更新全局汇总

#### 4-A：写入页面同目录的 live-spec.ts

在目标页面的**同目录**下创建或覆盖 `live-spec.ts`：

```ts
// 由 live-spec skill 自动生成，请勿手动编辑
// 页面：[页面名称]
// 更新时间：[当前日期]

export const pageName = '[页面中文名]'

export const spec = `
【[页面中文名]】
========================================
[生成的需求文档内容]
========================================
`
```

**格式强制检查**：写入前检查现有 `live-spec.ts` 中的 `spec` 字符串：
- 若已有内容但**不包含** `========================================`，说明是旧格式，必须按新格式完整重写
- 每次写入都必须确保 spec 字符串严格遵循以下结构，不得省略任何部分：
  1. 第一行：`【页面中文名】`
  2. 第二行：`========================================`
  3. 中间：需求文档内容
  4. 最后行：`========================================`

#### 4-B：更新全局汇总文件 live-spec.all.ts

在项目根目录（与 `package.json` 同级）维护一个 `live-spec.all.ts`，
汇总**所有页面**的需求文档，供「复制全部」功能使用。

每次任意页面的 `live-spec.ts` 被写入或更新后，**必须同步重新生成**此文件：

1. 扫描 `app/` 目录下所有已存在的 `live-spec.ts` 文件
2. 按页面路径排序
3. 输出汇总文件：

```ts
// 由 live-spec skill 自动生成，请勿手动编辑
// 更新时间：[当前日期]
// 共 N 个页面

export const allSpecs = `
【页面名1】
========================================
[页面1的 spec 内容]
========================================

【页面名2】
========================================
[页面2的 spec 内容]
========================================
`
```

4. 在 `components/LiveSpec.tsx` 的 import 处补充引用（若尚未引入）：

```tsx
import { allSpecs } from '@/live-spec.all'
```

并将 `allSpecs` 传入组件：

```tsx
<LiveSpec content={spec} pageName={pageName} allSpecs={allSpecs} />
```

> **注意**：若 `LiveSpec.tsx` 尚不接受 `allSpecs` prop，
> 则在组件 interface 中补充 `allSpecs?: string`，
> 并将「复制全部」按钮的复制内容从 `getAllSpecs()` 改为直接使用 `allSpecs ?? content`。

---

### Step 5：在页面中挂载悬浮球

检查目标 `page.tsx` 是否已引入 `<LiveSpec>`：

**若未引入**，在文件顶部添加 import，并在 JSX 根元素内末尾插入组件：

```tsx
import LiveSpec from '@/components/LiveSpec'
import { spec, pageName } from './live-spec'
import { history } from './live-spec.history'

// 在 return 的根元素末尾添加：
<LiveSpec content={spec} pageName={pageName} history={history} />
```

**若已引入**，确认 import 路径包含 `live-spec.history`，无需重复添加。
若 `live-spec.history.ts` 尚不存在（首次挂载），则 `history` import 暂时省略，
待首次 `/live-spec tag` 后自动生成再补充。

### Step 6：输出同步摘要

```
✅ live-spec 已同步

📄 需求文档：app/[路径]/live-spec.ts
🧩 LiveSpec 组件：components/LiveSpec.tsx（已存在 / 已新建）
📌 挂载位置：app/[路径]/page.tsx

启动 dev server 后，右下角悬浮球 📋 即可查看需求文档。
```

## 增量更新规则

当用户修改了页面的部分结构（如新增筛选字段、修改表格列），
**只更新 `live-spec.ts` 中受影响的条目**，不重新生成整个文档，
保留用户可能手动调整过的其他内容。

判断依据（同样融合三路信息源）：
- 本轮对话中用户提到的新需求 → 优先新增或修改对应条目
- 新增 JSX 元素 → 在对应层级新增编号条目
- 删除 JSX 元素 → 删除对应编号条目
- 修改交互逻辑 → 更新对应条目的描述行

更新后同样执行 Step 3 的自检清单，确保修改部分的缩进、空行、【】标注符合规范。

## 批量模式（/live-spec all）

1. 扫描 `app/` 目录下所有含 `page.tsx` 的子目录
2. 对每个页面依次执行 Step 2-3，生成各页面的 `live-spec.ts`
3. **格式迁移**：检查每个页面现有的 `live-spec.ts`，若 spec 字符串缺少 `========================================` 包裹，无论内容是否变化，强制按新格式重写
4. 统一执行一次 Step 4-B，生成完整的 `live-spec.all.ts`
5. 对每个页面执行 Step 5，挂载悬浮球
6. 输出汇总摘要，列出所有处理过的页面路径及状态（新建 / 已更新 / 格式迁移）

## 版本管理（/live-spec tag）

### 触发命令

| 命令 | 说明 |
|------|------|
| `/live-spec tag` | 扫描所有页面，为有变更的页面打版本快照 |
| `/live-spec tag "备注"` | 同上，附加备注文字显示在版本号旁 |

### 执行流程

#### Step T1：检测各页面变更

扫描 `app/` 目录下所有存在 `live-spec.ts` 的页面，逐一与该页面
`live-spec.history.ts` 中的最新版本 spec 做逐行对比：

- **无 history 文件**：视为首次打 tag，标记为「新建」
- **有 history 文件**：与最新版本 spec 逐行对比，统计变更条数和层级

变更统计维度：
- 新增行数（`+`）
- 删除行数（`-`）
- 修改行数（`~`，同编号描述文字发生变化）
- 是否涉及一级编号结构调整

#### Step T2：判断是否跳过

满足以下任一条件则跳过该页面，不新增版本：
- 当前 spec 与最新版本完全一致（无任何变更）
- 该页面尚无 `live-spec.ts`（尚未生成需求文档）

#### Step T3：推荐版本号并请用户确认

对所有**有变更**的页面，根据以下规则推荐版本号：

| 改动情况 | 推荐版本号 |
|---------|-----------|
| 首次打 tag | `v1` |
| 变更条目 ≤ 3 条，且均为二级及以下 | 当前大版本号 + `.1`（如 `v1.1`） |
| 变更条目 > 3 条，或涉及一级结构调整 | 大版本号 +1（如 `v2`） |

输出交互摘要，格式如下：

```
检测到 N 个页面有变更：

① 【页面名A】  新增 2 条，修改 1 条  →  建议 v1.1
② 【页面名B】  一级结构调整，新增 5 条  →  建议 v2
③ 【页面名C】  删除 1 条  →  建议 v1.1

无变更页面（已跳过）：【页面名D】【页面名E】

请确认各页面版本号（直接回复「确认」接受建议，或指定某页面的版本号）：
```

等待用户回复后，继续执行 Step T4。

#### Step T4：写入 live-spec.history.ts

在各有变更页面的同目录下创建或追加 `live-spec.history.ts`：

```ts
// 由 live-spec skill 自动生成，请勿手动编辑

export interface SpecVersion {
  version: string      // 如 "v1.1"
  label: string        // 如 "v1.1 · 2025-03-25 14:30"
  note: string         // 用户附加备注，无则为空字符串
  createdAt: string    // ISO 时间字符串
  spec: string         // 该版本的完整 spec 字符串（含页面名标题和分隔线）
}

export const history: SpecVersion[] = [
  {
    version: 'v1',
    label: 'v1 · 2025-03-25 14:30',
    note: '',
    createdAt: '2025-03-25T14:30:00.000Z',
    spec: `
【页面名】
========================================
[该版本完整 spec 内容]
========================================
`,
  },
  // 新版本追加在数组末尾，index 越大越新
]
```

**追加规则**：新版本始终 push 到数组末尾，不修改已有条目。

#### Step T5：输出打 tag 结果摘要

```
✅ live-spec tag 完成

① 【页面名A】  v1.1 · 2025-03-25 14:30
```
